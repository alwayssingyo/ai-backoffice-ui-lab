# Workflow Pipeline

## 목적
- 에이전트가 `web/` 작업을 수행할 때 따라야 하는 실행 순서를 고정합니다.
- 이 문서는 흐름과 게이트만 다룹니다.
- 어떤 규칙을 적용할지는 `rules.md`, 규칙 의미는 `source/*.md`를 따릅니다.

## 순서

### Stage 1. 요청 정리
- 목표, 제약, 완료조건, 변경 표면을 정리합니다.
- 완료조건은 확인 가능한 문장으로 정리합니다.

### Stage 2. 적용 규칙 선택
- `rules.md`에서 이번 변경의 `파일 표면 세트`와 `작업 단계 세트`를 각각 고릅니다.
- 최종 적용 규칙은 두 세트의 합집합으로 확정합니다.
- 원문은 필요한 분야만 읽습니다.
- 애매한 규칙만 해당 섹션을 다시 읽습니다.

### Stage 3. 컨텍스트 수집
- 관련 코드, 테스트, 기존 구현 패턴을 확인합니다.
- 변경 근거가 되는 파일 경로를 확보합니다.

### Stage 4. 구현
- 필요한 파일만 수정합니다.
- 생성 산출물은 직접 수정하지 않습니다.
  - `src/routeTree.gen.ts`
  - `src/services/**` 생성물
  - 생성되는 i18n/type/style/image 산출물

### Stage 5. 검증
- 항상 실행합니다.
  - `npm run build`
  - `npm run biome:check:all`
- 변경 동작에 따라 범위 검증을 추가합니다.
  - `npm run test`
  - `npm run test:e2e`
  - 범위 테스트

### Stage 6. 규칙 감사
- Stage 2에서 확정한 적용 규칙을 기준으로 변경 파일을 점검합니다.
- 자동 검사로 확인된 것과 수동으로 확인한 것을 구분합니다.
- `high` 위반이 있으면 Stage 4로 돌아갑니다.

### Stage 7. 최종 보고
- 무엇을 바꿨는지 설명합니다.
- 적용한 규칙 세트와 실행한 검증을 적습니다.
- 남은 리스크와 미실행 항목을 구분합니다.

## 게이트
- 적용 규칙을 고르기 전에는 구현하지 않습니다.
- 검증 없이 완료 처리하지 않습니다.
- `high` 위반을 문서화만 하고 종료하지 않습니다.

## 예외 규칙
- 컨벤션 예외는 기본 금지입니다.
- 예외가 필요하면 아래를 모두 만족해야 합니다.
  - 규칙 준수가 물리적으로 불가능합니다.
  - 대체안보다 리스크가 낮습니다.
  - 범위가 한 지점으로 제한됩니다.
- 코드 인접 주석 형식:
  - `// convention-exception: [.workflow/source/<file>.md#<section>] <사유>. TODO(<제거조건>).`

## 최종 응답 체크
- 적용한 원문 문서를 적었는가
- 적용한 `파일 표면 세트`와 `작업 단계 세트`를 적었는가
- 실행한 명령과 결과를 적었는가
- 자동 검사와 수동 감사를 구분했는가
- 남은 리스크를 적었는가
- 예외가 있으면 근거와 제거 조건을 적었는가
